## 문제 정의

중전기 사업부는 매월 사업계획 대비 수주·매출 실적을 비교하는 보고서를 작성한다.  
기존 프로세스에는 세 가지 핵심 문제가 있었다.

| 문제 | 영향 |
|------|------|
| **4개 파일(400~900행) 수작업 대조** | 1회 작성에 4~6시간 소요, 오류 발생 위험 |
| **프로젝트 단위 추적 불가** — 합계 숫자만 비교 | 차이가 *얼마인지*는 보이지만 *왜 생겼는지*를 알 수 없음 |
| **계획↔실적 파일 간 프로젝트명 불일치** | 같은 프로젝트가 다른 이름·코드로 표기돼 매칭 누락 |

실제로 ₩300억 규모의 해외 수주건이 1분기→5월로 이월됐음에도  
합계 대조 방식에서는 식별되지 않은 채로 보고서가 작성된 사례가 있었다.

---

## 해결 방법

4개 CSV → 자동 매칭 → 원인 분류 → Excel 보고서 + 차트 생성까지  
전 과정을 Python으로 자동화한 파이프라인.

### 5단계 프로젝트 매칭 캐스케이드

사람이 프로젝트를 확인하는 방식을 그대로 코드로 구현했다.  
신뢰도가 높은 기준부터 순서대로 적용하고, 실패하면 다음 단계로 내려간다.

```
1순위 → 프로젝트명 완전일치       (가장 확실한 기준)
2순위 → PJT 번호 일치            (시스템 부여 ID)
3순위 → 발주처(EPC) 일치         (같은 제품군 내)
4순위 → 원청사 일치
5순위 → 퍼지 매칭 (유사도 ≥ 80%) (명칭 변경 대응)
```

퍼지 매칭 기준 80%는 실제 데이터로 실험하며 결정했다.  
`rapidfuzz.token_sort_ratio`는 단어 순서 차이(예: "1단계 알파" vs "알파 1단계")를 무시하므로  
명칭 표기 방식이 달라도 올바르게 매칭된다.

- 계획에만 있는 건 → **소멸** (취소·이월 추정)
- 실적에만 있는 건 → **신규** (계획외 수주)

## 1분기 분析 결과

| 항목 | 계획 | 실적 | 차이 | 달성률 |
|------|-----:|-----:|-----:|-------:|
| **중전기 수주** | 2,607.5억 | 4,290.1억 | **+1,682.6억** | **164.5%** |
| — 변압기 | 2,231.4억 | 3,875.0억 | +1,643.6억 | 173.7% |
| — 차단기 | 376.2억 | 415.0억 | +38.8억 | 110.3% |
| **중전기 매출** | 1,140.5억 | 1,214.3억 | +73.8억 | 106.5% |
| — 변압기 | 906.7억 | 974.2억 | +67.5억 | 107.4% |
| — 차단기 | 233.8억 | 240.1억 | +6.3억 | 102.7% |

**프로젝트 단위 매칭 결과:**

| 분류 | 수주 | 매출 |
|------|-----:|-----:|
| 매칭 (계획↔실적 연결) | 38건 | 41건 |
| 소멸 (계획에만 존재) | 11건 | 7건 |
| **신규 (실적에만 존재)** | **68건** | 52건 |

계획 대비 164.5% 달성의 핵심 원인은 계획에 없던 **해외 신규 수주 68건**이었으며,  
미국 풍력·태양광 단지 관련 건이 다수를 차지했다.

**보고서-CSV 수치 불일치 추적 완료:**  
중간 보고서와 CSV 집계값 사이의 308억 차이를 항목별로 전부 추적했다.  
해외 1개 프로젝트(300억)의 확정월이 1분기→5월로 변경된 것이 주요 원인이었고  
나머지 8억은 팀별 소규모 조정에서 비롯됐다.

## 파이프라인 구조

```
입력 CSV 4개
┌─────────────────────────────────────────┐
│  수주 계획      수주 실적               │
│  매출 계획      매출 실적               │
└──────────────┬──────────────────────────┘
               │
               ▼
         [분석.py — 메인]
               │
   ┌───────────┼────────────┐
   ▼           ▼            ▼
Step 1       Step 2      Step 3
로드/전처리   요약 비교   프로젝트 매칭
               │            │
           월×팀×제품군   5단계 캐스케이드
           계획vs실적     → 매칭/신규/소멸
   └───────────┴────────────┘
               │
               ▼
         [Step 4 — 결과물 생성]
               │
   ┌───────────┼────────────┐
   ▼           ▼            ▼
charts/    차이分析        텍스트 사유
(PNG 7종)  결과.xlsx      생성
           (7개 시트)
               │
               ▼
      [차이원인分析.py] → 차이원인分析.xlsx
               │
               ▼
      [양식생성.py]    → 월별차이分析_1분기.xlsx
```

## 기술 스택

| 라이브러리 | 버전 | 선택 이유 |
|-----------|------|----------|
| **pandas** | ≥1.5 | groupby 집계 + outer merge로 계획/실적 비교. Excel VBA 대비 압도적으로 빠름 |
| **numpy** | ≥1.23 | NaN 처리 및 달성률 조건부 계산 |
| **chardet** | ≥5.0 | 소스 파일이 EUC-KR·UTF-8 혼재. 파일별 자동 감지로 수동 설정 제거 |
| **rapidfuzz** | ≥3.0 | `token_sort_ratio`로 단어 순서 무시 매칭. 순수 Python `difflib` 대비 수십 배 빠름 |
| **matplotlib** | ≥3.6 | 워터폴·막대·도넛 차트 코드 생성 — Excel 차트 수작업 없음 |
| **openpyxl** | ≥3.1 | 셀 단위 서식(조건부 색상, 병합, 테두리)으로 보고서 품질 Excel 출력 |

## 기술적 도전과 해결

| 문제 | 원인 | 해결 방법 |
|------|------|----------|
| **수주 계획 파일 상단 9행이 쓰레기 데이터** | 소스 시스템이 환율·구성비 메타데이터를 헤더 앞에 삽입 | `skiprows=list(range(9)), header=0` — 원본 파일 바이트 분석으로 발견 |
| **매출 실적 파일 상단 4행이 환율 헤더** | 다른 모듈의 별도 내보내기 템플릿 | 해당 파일에만 `skiprows=list(range(4))` 선택적 적용 |
| **팀명 형식 불일치** (`01.국내1` vs `국내1`) | 계획 시스템이 팀 코드 앞에 번호 붙임 | `re.sub(r'^\d+\.', '', name)` 정규식 정규화 |
| **제품군 표기 혼재** (계획: `변압기`, 실적: `TR`) | 계획/운영 시스템이 서로 다른 코드 체계 사용 | 단일 `classify_prod()` 함수에서 양쪽을 모두 정규 명칭으로 변환 |
| **프로젝트명 표기 드리프트** | 데이터 입력 시 공유 PJT ID 미강제, 프로젝트명이 시간에 따라 변경 | 5단계 캐스케이드 + rapidfuzz 퍼지 매칭 (≥80%) |
| **보고서-CSV 308억 불일치** | 보고서 기준일 이후 프로젝트 확정월 변경 | 팀별 세부 내역 추적으로 항목별 확인 완료 |
| **원본 Excel DRM 보호** | 사내 DRM 정책으로 직접 쓰기 불가 | CSV 내보내기 경로로 우회, write-back 실패 시 수동 입력값 출력 |

## 실행 방법

### 패키지 설치

```bash
pip install -r requirements.txt
```

### 샘플 데이터로 실행 (기본값)

```bash
# Step 1 — 메인 분析 (SAMPLE_MODE = True가 기본값)
python 분析.py

# Step 2 — 원인 분류 상세 분析
python 차이원인分析.py

# Step 3 — 월별 보고서 양식 생성
python 양식생성.py
```

결과 파일은 프로젝트 루트와 `charts/` 폴더에 생성된다.

### 실제 데이터 사용 시

1. `csv/` 폴더에 실제 CSV 파일 배치
2. `분析.py` 상단 `SAMPLE_MODE = False` 로 변경 (약 35번째 줄)
3. `FILE_*` 변수 4개가 실제 파일명과 일치하는지 확인
4. 동일하게 3개 스크립트 순서대로 실행

### 샘플 데이터 구조

| 파일 | 행수 | 비고 |
|------|-----:|------|
| `csv/sample_사업계획_수주.csv` | 10행 | 상단 9행 헤더 포함 (실제 파일과 동일한 구조) |
| `csv/sample_실적_수주.csv` | 14행 | OUT 필터링 테스트용 행 포함 |
| `csv/sample_사업계획_매출.csv` | 12행 | 표준 CSV |
| `csv/sample_실적_매출.csv` | 12행 | 상단 4행 헤더 포함 (실제 파일과 동일한 구조) |

샘플에는 매칭 프로젝트, 소멸 1건(계획 200억, 실적 없음), 신규 3건이 포함되어  
세 가지 분류 결과를 모두 확인할 수 있다.

## 한계 및 향후 계획

**현재 한계**
- 1분기(1~3월) 전용으로 구현됨 — 4~7월 컬럼 구조는 준비됐으나 데이터 미입력
- 매출 실적 275억 불일치는 환율 적용 기준 차이(USD 1,400 vs 1,300)로 추정하나 미확정
- `보고기준_서식및매칭.py`의 팀별 소계가 하드코딩돼 있어 매 사이클마다 수동 수정 필요

**향후 개선 계획**
- [ ] 분기·월 파라미터화 (현재 1분기 하드코딩)
- [ ] 매칭 캐스케이드 엣지케이스 단위테스트 추가
- [ ] 팀별 소계 하드코딩 → CSV 동적 집계로 전환
- [ ] 반기·연간 보고서 양식 확장
- [ ] CLI 진입점 추가 (`python run.py --quarter Q2 --mode sample`)
